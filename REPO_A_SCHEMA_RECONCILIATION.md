## 1. Canonical Repo A Persistence Model

The following is the exact persistence model as implemented in `lsg_ingest.html → writeToSupabase()` and `lsg_one_pager.html → buildSyncRow()`. This is not the master doc description — it is derived directly from the working code.

### Table name

```
deals
```

Referenced as the constant `DEALS_TABLE = 'deals'`. All REST calls, PATCHes, and SELECTs use this name. Cannot be changed without updating all Supabase REST URL constructions in both HTML files.

---

### Exact row written on ingest (from `writeToSupabase()`, `lsg_ingest.html` line 1411)

```
Column name       Type in Supabase   Source in code
─────────────────────────────────────────────────────────────────────────────
id                UUID               generated by Supabase DEFAULT gen_random_uuid()
ingested_at       TIMESTAMPTZ        data._ingested_at || new Date().toISOString()
deal_name         TEXT               data.deal_name
address           TEXT               [data.address, data.city, data.state].filter(Boolean).join(', ')
asset_type        TEXT               data.property?.type
purchase_price    NUMERIC            parseAmt(data.sources_uses?.purchase_price)
going_in_cap      NUMERIC            parsePct(data.returns?.going_in_cap)
irr_levered_5     NUMERIC            parsePct(data.returns?.irr_levered_5)
moic_5            NUMERIC            parseFloat(String(data.returns?.moic_5).replace(/x/gi,''))
status            TEXT               data.transaction?.status || 'Screening'
source_broker     TEXT               data.transaction?.sourcing
assignee          TEXT               data._assignee
source_files      TEXT[]             data._source_files (array of filenames)
schema_version    TEXT               data.schema_version || '2.0'
raw_data          JSONB              full extracted JSON (~200 fields), with _reader_narrative deleted
```

That is **15 columns** as the canonical write payload. The `id` column is generated by the database; the remaining 14 are written by application code.

---

### `raw_data` is the source of truth

`raw_data` is a JSONB column containing the complete extracted JSON object from schema v2.0. It holds approximately 200 fields across 25 top-level objects. The scalar columns above (`purchase_price`, `going_in_cap`, etc.) are **derived index values only** — they are parsed subsets of what is already in `raw_data`. If `raw_data` and a scalar column disagree, `raw_data` is correct.

Any feature that displays deal data must read from `raw_data`. Scalar columns exist only to support filtering and sorting the pipeline table without loading the full JSONB object.

---

### Naming convention

**snake_case throughout.** Every column name, every field accessed via Supabase REST URL param, every key in the row object written from `writeToSupabase()`. This is not a convention choice — it is hardcoded into the function bodies. The row object `{ deal_name: ..., asset_type: ..., purchase_price: ..., going_in_cap: ..., irr_levered_5: ..., source_broker: ... }` is sent directly as the POST/PATCH body.

---

### How `_deal_id` works

After a successful Supabase POST, the returned UUID is extracted:
```
savedId = Array.isArray(saved) ? saved[0]?.id : saved?.id
```

This UUID is written back to the extracted JSON object as `extracted._deal_id`. It is then:
1. Stored in `localStorage` as part of the full JSON (`lsg_op_data_v1`)
2. Used by the one-pager's `syncToDB()` to route: if `_deal_id` exists → PATCH existing row, if not → POST new row
3. Passed to the inline edit panel to route its PATCH correctly

`_deal_id` is therefore the runtime linkage key between the extracted JSON object in memory/localStorage and the Supabase row. It is never a Supabase column — it lives only inside `raw_data` and in the in-memory JSON object.

---

### `screen` — status and meaning

`screen` is patched to the `deals` table as a top-level column by `runScreen()` in `lsg_ingest.html` (line 2033):
```
body: JSON.stringify({ screen: result })
```

The `result` object is the AI screener's JSON output: verdict (PASS/WATCH/FAIL), metric cards, risk flags, next steps, basis sensitivity, TIF note, ran_at timestamp.

**`screen` is not in the documented CREATE TABLE SQL.** It is either:
- A column that was added to the live Supabase instance outside the documented schema, or
- A PATCH that fails silently (the catch block at line 2035 logs a warning and continues without error)

Evidence for the second interpretation: `loadDeals()` at line 1559 does not include `screen` in its SELECT clause. This means screen results are not loaded on table refresh — they only persist in the in-memory `allDeals` array within the same browser session.

**Current conclusion:** Screen results are effectively ephemeral in Repo A today. The `screen` column either does not exist in the live table or is not being read back. This is a known gap, not an established pattern to replicate.

---

### Whether `extraction_meta`, `conflict_flags` must be in the DB

**No.** These fields are stored inside `raw_data` JSONB, not as separate columns. They are accessed from `raw_data` at display time:
- `d.extraction_meta?.confidence`, `d.extraction_meta?.fields_populated`
- `(d.conflict_flags || []).filter(c => c.field)`

There is no application code that reads `extraction_meta` or `conflict_flags` as top-level Supabase columns. They do not need to exist outside of `raw_data`.

## 2. Minimum Schema for Merged v1

### Must-have now (required for any end-to-end testing to be meaningful)

```sql
-- The 14 application-written columns in snake_case
id               UUID PRIMARY KEY DEFAULT gen_random_uuid()
ingested_at      TIMESTAMPTZ DEFAULT now()
deal_name        TEXT
address          TEXT
asset_type       TEXT
purchase_price   NUMERIC
going_in_cap     NUMERIC
irr_levered_5    NUMERIC
moic_5           NUMERIC
status           TEXT DEFAULT 'Screening'
source_broker    TEXT
assignee         TEXT
source_files     TEXT[]
schema_version   TEXT
raw_data         JSONB   -- NON-NEGOTIABLE

-- RLS must be disabled (or a permissive policy added)
ALTER TABLE deals DISABLE ROW LEVEL SECURITY;
GRANT ALL ON TABLE deals TO anon;
GRANT USAGE ON SCHEMA public TO anon;
```

`raw_data JSONB` is the single non-negotiable item. Without it, no extraction output can be stored. The merged product is non-functional from a data perspective without this column.

The 4 NUMERIC columns (`purchase_price`, `going_in_cap`, `irr_levered_5`, `moic_5`) must be NUMERIC, not TEXT. Repo A's write code passes parsed numbers, not strings, for these fields. Inserting a number into a TEXT column may succeed in Postgres (implicit cast) but will break sort-by-price, filter-by-cap-rate, and any arithmetic performed on these columns.

---

### Can wait until Phase 2 or Phase 3

```
screen           JSONB    -- screener results; currently ephemeral anyway
city             TEXT     -- currently collapsed into address scalar
state            TEXT     -- currently collapsed into address scalar
created_at       TIMESTAMPTZ  -- audit column; useful but not required for v1
updated_at       TIMESTAMPTZ  -- audit column; useful but not required for v1
```

Indexes on `status`, `asset_type`, `ingested_at` can wait until query performance is a concern (typically > 1,000 rows).

## 3. Comparison Against Repo B's Current Temporary Schema

**Working assumption:** Repo B's 18 columns are a camelCase version of the Repo A scalar set, likely approximating: `id`, `createdAt`/`updatedAt`, `dateAdded`, `dealName`, `address`, `assetType`, `purchasePrice`, `goingInCap`, `irrLevered5`, `moic5`, `status`, `sourceBroker`, `assignee`, `sourceFiles`, `schemaVersion`, plus city, state, or other fields derived from reading the master doc. Exact column list not provided and not invented below.

| Difference | Classification | Reason |
|---|---|---|
| No `raw_data JSONB` column | **Must change before Milestone 1 testing** | Without this, every extraction write fails or produces an incomplete record. All end-to-end testing without it is testing a broken pipeline. Cannot defer. |
| camelCase column names vs. snake_case | **Must change before Milestone 1 testing** | Repo A's `writeToSupabase()` sends `{ deal_name, asset_type, purchase_price, going_in_cap, irr_levered_5, source_broker }` as the POST body. If the table has `dealName`, `assetType`, etc., Supabase returns a 400 (column not found) or silently ignores the fields. The write will not produce correct rows. |
| TEXT for `purchasePrice` / `goingInCap` / `irrLevered5` / `moic5` | **Must change before Milestone 1 testing** | Repo A sends parsed numbers for these fields (output of `parseAmt()`, `parsePct()`, `parseFloat()`). Writing a number to a TEXT column may succeed via implicit cast in Postgres, but the pipeline table sort-by-price and hurdle-rate comparisons require NUMERIC. Fixing after data is stored means migrating existing rows. Fix the type before writing any real data. |
| `source_files TEXT` (single value) vs. `TEXT[]` (array) | **Must change before Milestone 1 testing** | Repo A writes `source_files: data._source_files || []` — an array. Inserting an array into a TEXT column fails. If Repo B stores source files as a single TEXT string, insertion will throw a type error unless the application silently converts. |
| `dateAdded TEXT` vs. `ingested_at TIMESTAMPTZ` | **Must change before Milestone 1 testing** | Repo A writes `ingested_at: data._ingested_at || new Date().toISOString()`. If the table has `dateAdded TEXT` and no `ingested_at`, the timestamp is discarded. Worse, `dateAdded TEXT` cannot be sorted chronologically without string-date parsing. |
| `created_at` and `updated_at` auto-managed columns | **Acceptable, keep** | These are standard and useful. Repo A doesn't write them (they default to `now()`) so they don't conflict. Keep them in the merged product. |
| UUID `id` column | **Acceptable, keep** | Repo A expects a UUID primary key named `id`. This matches. |
| 18 scalar TEXT columns (no JSONB) covering Repo A's scalar set | **Acceptable as a starting point for column list** | The column set is approximately right; the problem is types and naming convention, not column selection. |
| No `screen` JSONB column | **Acceptable for now** | Screen results are currently ephemeral in Repo A too. This is not a regression. Add it in Phase 2 when screener persistence is a tested requirement. |
| Extra columns in Repo B beyond Repo A's schema | **Not important** | Additional columns Repo B added for its own Milestone 1 needs are fine. They won't conflict with Repo A's writes. |

## 4. Recommended Decision

### Decision: C — Hybrid. Specific fields now; remainder deferred.

**What must change now (before any end-to-end testing):**

1. Add `raw_data JSONB`
2. Rename all camelCase columns to snake_case for the Repo A scalar set
3. Change `purchasePrice/goingInCap/irrLevered5/moic5` from TEXT to NUMERIC
4. Change `sourceFiles` from TEXT to `TEXT[]`
5. Rename `dateAdded TEXT` to `ingested_at TIMESTAMPTZ` (or add `ingested_at TIMESTAMPTZ` and keep `dateAdded` as deprecated)

**What can stay as-is:**

- UUID `id` (correct)
- `created_at`, `updated_at` (keep; useful additions Repo B made)
- `status TEXT DEFAULT 'Screening'` (probably fine if naming matches)
- Any extra columns Repo B added that don't conflict

**What should be deferred:**

- `screen JSONB` — Phase 2
- `city TEXT`, `state TEXT` as separate columns — Phase 2 (requires changes to Repo A's address construction logic)
- Indexes on status, asset_type, ingested_at — Phase 3 (when table is large enough to matter)

---

### Why not Option A (keep Repo B schema for Milestone 1, migrate later)?

Option A fails for a fundamental reason: the missing `raw_data JSONB` column is not a compatibility problem that a migration solves later — it means **no extraction data is stored during Milestone 1 testing.** Every test run produces a record with only the 14 scalar fields and no deal content. When the team declares "end-to-end works," it is testing a hollow pipeline. The schema migration will require discarding all test data collected during Milestone 1. This is avoidable waste.

The camelCase naming issue compounds this: with camelCase table columns, Repo A's writes will either fail (400 from Supabase) or silently store nothing if a column mapper is introduced. Either way, Milestone 1 testing is measuring a broken integration path.

---

### Why not Option B (full replacement now)?

Option B is overkill. Repo B's column selection is approximately right — it identified the right scalar fields from the master doc. The problems are correctable in place: add one column, rename column names, change four types. A full schema replacement would discard the work Repo B did correctly and force a rebuild that takes longer than a targeted fix.

---

### Why Option C is right

The five changes listed above are all surgical. None requires a full table rebuild. In Postgres with Supabase:
- Adding a JSONB column: `ALTER TABLE deals ADD COLUMN raw_data JSONB;`
- Renaming columns: `ALTER TABLE deals RENAME COLUMN dealName TO deal_name;`
- Changing TEXT to NUMERIC: `ALTER TABLE deals ALTER COLUMN purchase_price TYPE NUMERIC USING purchase_price::NUMERIC;`
- Changing TEXT to TEXT[]: `ALTER TABLE deals ALTER COLUMN source_files TYPE TEXT[] USING ARRAY[source_files];`
- Renaming/adding ingested_at: one rename or one new column with backfill

This is approximately 10 SQL statements. It should be executed against the Supabase table before Milestone 1 testing begins. The result is a schema that is correct by Repo A authority standards and keeps what Repo B built correctly.

---

### On the mapper layer question

A camelCase ↔ snake_case mapper at the Vercel API route layer is technically possible — the route receives camelCase from React and translates to snake_case before writing to Supabase. This is **not recommended** for the persistence layer for two reasons:

1. It creates a hidden translation step that will confuse future developers and cause bugs when one side of the map is updated and the other is not.
2. It defers the inevitable. The mapper will need to be removed when the schema is corrected, at which point the mapping code, tests, and documentation all need cleanup. The correction is 10 SQL statements today; cleaning up a mapper is harder.

A mapper is acceptable for the **presentation layer** — transforming `deal_name` from the database into `dealName` for React component props is fine and idiomatic. It is not acceptable as a substitute for a correct database schema.

## 5. Exact Migration Implications

The following is the exact change list for Repo B's Supabase table. Assumes no live production data exists that would be lost — this is a development/Milestone 1 schema correction, not a migration of production records.

---

### Must add (no equivalent exists in Repo B)

```sql
ALTER TABLE deals ADD COLUMN raw_data JSONB;

-- Required: array of source document filenames

```

### Must rename (camelCase → snake_case)

The exact SQL depends on Repo B's actual column names. The expected renames based on the Repo A scalar set are:

```sql
-- Apply only the renames that correspond to mismatched Repo A columns.
-- Do not rename columns that are Repo B-specific and not in Repo A's write payload.
ALTER TABLE deals RENAME COLUMN dealName TO deal_name;
ALTER TABLE deals RENAME COLUMN assetType TO asset_type;
ALTER TABLE deals RENAME COLUMN purchasePrice TO purchase_price;
ALTER TABLE deals RENAME COLUMN goingInCap TO going_in_cap;
ALTER TABLE deals RENAME COLUMN irrLevered5 TO irr_levered_5;
ALTER TABLE deals RENAME COLUMN moic5 TO moic_5;
ALTER TABLE deals RENAME COLUMN sourceBroker TO source_broker;
ALTER TABLE deals RENAME COLUMN schemaVersion TO schema_version;
-- If sourceFiles exists as camelCase TEXT:
ALTER TABLE deals RENAME COLUMN sourceFiles TO source_files;
```

Verify the exact column names in Repo B's table before running. Rename only columns that exist and conflict.

---

### Must change type (TEXT → NUMERIC)

```sql
-- Execute after any renames above so the column names are correct.
ALTER TABLE deals ALTER COLUMN purchase_price TYPE NUMERIC USING purchase_price::NUMERIC;
ALTER TABLE deals ALTER COLUMN going_in_cap   TYPE NUMERIC USING going_in_cap::NUMERIC;
ALTER TABLE deals ALTER COLUMN irr_levered_5  TYPE NUMERIC USING irr_levered_5::NUMERIC;
ALTER TABLE deals ALTER COLUMN moic_5         TYPE NUMERIC USING moic_5::NUMERIC;
```

The `USING` clause handles the TEXT → NUMERIC cast for any existing rows. Rows with non-numeric text in these columns will fail the cast — handle by setting to NULL first if any test data exists.

---

### Must rename or add (timestamp handling)

```sql
-- Option A: rename dateAdded to ingested_at and change type
ALTER TABLE deals RENAME COLUMN dateAdded TO ingested_at;
ALTER TABLE deals ALTER COLUMN ingested_at TYPE TIMESTAMPTZ USING ingested_at::TIMESTAMPTZ;
ALTER TABLE deals ALTER COLUMN ingested_at SET DEFAULT now();

-- Option B: add ingested_at alongside dateAdded (keep dateAdded for Repo B compatibility)
ALTER TABLE deals ADD COLUMN ingested_at TIMESTAMPTZ DEFAULT now();
```

Option A is cleaner. Option B is acceptable if Repo B has significant code referencing `dateAdded` that can't be updated immediately.

---

### Should remain unchanged (keep as-is)

```sql
id          UUID PRIMARY KEY DEFAULT gen_random_uuid()   -- correct as-is
created_at  TIMESTAMPTZ DEFAULT now()                    -- keep; useful audit column
updated_at  TIMESTAMPTZ                                  -- keep; useful audit column
status      TEXT DEFAULT 'Screening'                     -- keep if naming is already snake_case
assignee    TEXT                                         -- keep if already snake_case
address     TEXT                                         -- keep; Repo A flattens city/state into this
```

Any Repo B-specific columns beyond the Repo A scalar set may stay. They won't conflict with Repo A writes unless their names collide with the write payload keys above.

---

### Must also verify (RLS and grants)

```sql
-- Required for Supabase REST API to accept writes without auth
ALTER TABLE deals DISABLE ROW LEVEL SECURITY;
GRANT ALL ON TABLE deals TO anon;
GRANT USAGE ON SCHEMA public TO anon;
```

If Repo B's Supabase instance has RLS enabled or different grant configuration, Repo A's writes will return 403. This is a common silent failure — the write appears to succeed but returns a permission denied error that Repo A translates as "disable RLS."

---

### Can defer to Phase 2 or Phase 3

```sql
-- Phase 2: screener result persistence
ALTER TABLE deals ADD COLUMN screen JSONB;

-- Phase 2: granular location columns (requires updating Repo A address construction)
ALTER TABLE deals ADD COLUMN city TEXT;
```
